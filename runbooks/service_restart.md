# サービス再起動の共通手順

## 概要
マイクロサービスの計画的・緊急再起動の標準手順。

## 再起動の判断基準
以下のいずれかに該当する場合に再起動を検討:
- CPU使用率が90%超で継続（5分以上）
- メモリ使用率が85%超で増加中
- レイテンシが500ms超で継続
- エラー率が5%超で継続
- ヘルスチェックが `degraded` を返している
- ログにプロセス詰まりを示すパターンが出ている

## 再起動前チェックリスト

1. **影響範囲の確認**
   - 再起動対象サービスの現在のトラフィック量（RPS）を確認
   - 依存サービス（このサービスに依存しているもの）を確認
   - ピーク時間帯でないことを確認（可能な場合）

2. **理由の記録**
   - 再起動理由を明確に記録すること（監査ログに残る）
   - 例: "CPU 95%が5分間継続、ERRORログにDB接続タイムアウト多発"

3. **バックアップ計画**
   - 再起動後もサービスが起動しない場合の連絡先を確認

## 再起動手順

```
restart_service(
    service_name="[サービス名]",
    reason="[具体的な理由]"
)
```

## 再起動後の確認（必須）

1. **起動確認**: ステータスが `running` になることを確認
2. **ヘルスチェック**: `get_health(service)` が `healthy: true` を返すことを確認
3. **メトリクス正常化**: 5分後にメトリクスが正常値に戻ったことを確認
   - CPU: < 70%
   - メモリ: < 70%
   - レイテンシ: < 200ms
   - エラー率: < 1%
4. **ログ確認**: 再起動後のログに新たなエラーがないことを確認

## 再起動で解消しない場合

- スケールアップを試みる（別サービスインスタンスに負荷分散）
- 上流・下流サービスの問題を確認
- 開発チームにエスカレーション

## 注意事項

- `api-gateway` の再起動は全サービスへの影響が最大 → 慎重に判断
- ステートフルなサービスの場合、セッション情報が失われる可能性あり
- 再起動は最終手段；まずスケールアップを検討すること
